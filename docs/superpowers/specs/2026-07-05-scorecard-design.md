# 미너비니 정산표(Trading Scorecard) 설계 문서

- 작성일: 2026-07-05
- 근거: 마크 미너비니 『챔피언처럼 생각하고 거래하라』 2·3·4장 + 책의 "월별 확인표 / 트레이딩 요약" 스크린샷(그림 4-2, 130쪽)
- 관련 메모리: [[sepa-holdings-feedback]], [[plain-language-explanations]], [[doc-logic-sync]]

## 1. 목적과 배경

미너비니의 매매 철학에서 **진입 패턴(VCP·파워플레이·3C)**은 이미 이 저장소에 구현돼 있다(`find-*` 스킬). 그러나 2~4장의 핵심인 **"기록이 곧 진실 — 거래 결과를 정산표로 계산해 다음 거래를 지휘한다"**는 관리·기록 층은 비어 있다.

이 문서는 그 빈 조각, 즉 **정산표(Trading Scorecard)**를 만드는 설계다. 왕복거래별로 평균수익·평균손실·승률·손익비·기대수익·보유일수를 계산하고, 책의 월별 확인표와 트레이딩 요약을 그대로 재현한다.

### 중요한 범위 제약

- **원천 데이터는 기존 `journal.json`(173건)을 쓰지 않는다.** 그 173건 대다수는 미너비니 원칙대로 수행한 매매가 아니다. 책은 "전략이 다른 거래는 평균에 섞지 말라"고 못박았다.
- 정산표는 **미너비니 원칙을 지킨 거래만 담는 새 전용 장부**에서 계산한다. 데이터는 앞으로 사용자가 하나씩 추가하며 키운다(현재 소수 건).

## 2. 만드는 순서 (2단계)

- **C단계(먼저)**: 계산 로직(순수 모듈) + CLI 스크립트 + JSON 산출 + 터미널 표 출력. 데이터가 적어도 로직을 탄탄히 다진다.
- **A단계(다음)**: `/journal/scorecard` 전용 페이지가 산출된 JSON을 읽어 시각화. 계산 로직은 C단계 모듈을 그대로 재사용(로직 1벌).

이 문서는 **C단계의 완전한 스펙**이며, A단계는 개요만 담는다(별도 스펙에서 상세화).

## 3. 데이터 모델

### 3.1 입력: 체결 장부 `public/data/scorecard-fills.json`

미너비니 원칙대로 한 거래의 체결(매수/매도)을 시간순으로 쌓는다. 기존 `journal.json`과 완전히 분리된 별도 파일.

```json
{
  "strategy": "minervini",
  "stop_loss_pct_default": -4,
  "fills": [
    {
      "date": "2026-07-01 09:31:32",
      "code": "036800",
      "name": "나이스정보통신",
      "side": "buy",
      "price": 29700,
      "qty": 435,
      "fees": 1290,
      "tax": 0,
      "stop": 28512,
      "setup": "VCP",
      "note": "피벗 29,099 돌파 진입"
    }
  ]
}
```

필드 정의:

| 필드 | 필수 | 의미 |
|---|---|---|
| `date` | 예 | 체결 시각. `"YYYY-MM-DD HH:MM:SS"` 또는 `"YYYY-MM-DD"`. 월 귀속·보유일수 계산에 사용 |
| `code` | 예 | 종목 코드(6자리 문자열) |
| `name` | 예 | 종목명(표시용) |
| `side` | 예 | `"buy"` 또는 `"sell"` |
| `price` | 예 | 체결 단가(원) |
| `qty` | 예 | 체결 수량(주). 양의 정수 |
| `fees` | 아니오(기본 0) | 수수료(원). 순수익률 계산에 사용 |
| `tax` | 아니오(기본 0) | 세금(원, 보통 매도에만). 순수익률 계산에 사용 |
| `stop` | 아니오 | 진입 시 계획한 손절가(원). **통계엔 안 씀**, 규율 점검 전용 |
| `setup` | 아니오 | 셋업 태그(`"VCP"`/`"PowerPlay"`/`"3C"` 등). **지금 집계엔 안 씀**, 향후 패턴별 분리용 |
| `note` | 아니오 | 자유 메모 |

### 3.2 왕복거래 묶기 규칙 (`matchTrades`)

체결 목록을 **종목별로 시간순 처리**하며 포지션을 누적해 "왕복거래(round-trip)"로 확정한다.

- 종목의 누적 보유수량이 **0 → 양수**가 되는 순간 포지션이 **열린다**.
- 매수는 수량을 더하고, 매도는 뺀다(**분할 매수=피라미딩, 분할 매도=부분 익절/단계별 스톱**을 그대로 흡수).
- 누적 보유수량이 다시 **정확히 0**이 되면 그 포지션을 **하나의 왕복거래로 확정**한다.
- 종목을 청산 후 다시 매수하면 **별개의 새 왕복거래**로 카운트한다.
- 마지막 매도 후에도 수량이 **남아 있으면(미청산)** 그 포지션은 **"열린 거래"**로 따로 보관하고 **실현 통계에서 제외**한다(책: 실현된 결과만이 진실).

#### 엣지 케이스 규칙

- **과매도(수량 음수)**: 누적 보유수량이 음수로 내려가는 체결은 **데이터 오류**로 간주하고 경고를 출력하되, 그 종목은 통계에서 제외한다(공매도는 이 전략 범위 밖).
- **부분청산 후 재매수**: 포지션이 열린 상태에서 일부 매도(수량>0 유지) 후 다시 매수하면 **같은 왕복거래에 계속 흡수**한다. 오직 수량이 0이 될 때만 거래가 닫힌다.
- **같은 날 여러 체결**: 시간까지 있으면 시간순, 날짜만 있으면 파일 등장 순서를 tie-break로 사용(안정 정렬).

### 3.3 왕복거래 1건에서 계산하는 값

확정된 왕복거래마다 다음을 산출한다.

- **가중평균 매수가** = Σ(매수 price × qty) ÷ Σ(매수 qty)
- **가중평균 매도가** = Σ(매도 price × qty) ÷ Σ(매도 qty)
- **총수익률(gross)** % = (가중평균매도가 ÷ 가중평균매수가 − 1) × 100
- **순수익률(net)** %:
  - 순매수원가 = Σ(매수 price×qty) + Σ(매수 fees)
  - 순매도수취 = Σ(매도 price×qty) − Σ(매도 fees) − Σ(매도 tax)
  - 순수익률 = (순매도수취 ÷ 순매수원가 − 1) × 100
- **보유일수** = (마지막 매도일 − 첫 매수일)의 **역일수(calendar days)**. 같은 날이면 0.
- **승/패 분류**: **순수익률 > 0 이면 수익거래(win), 그 외 손실거래(loss)**. (정확히 0인 본전 거래는 손실 쪽으로 분류 — 미너비니 기준상 "이기지 못한 거래")
- **월 귀속**: **마지막 매도일(청산 완료 시점)의 연-월**로 귀속한다. 월별 확인표는 "매도 트레이딩"을 월별로 집계하므로 청산월 기준.
- **규율 점검(선택, `stop`이 있을 때만)**: 손실거래인데 순수익률(또는 총수익률)이 계획 손절폭보다 더 나빴으면 `stop_violation = true` 로 표시.

## 4. 지표 계산 (`computeScorecard`)

모든 % 지표는 **순수익 기준과 총수익 기준 두 벌**로 계산한다. CLI·페이지 표시 기본값은 **순수익**, 총수익은 나란히 병기.

### 4.1 두 가지 집계 방식 (스크린샷 검산으로 확정)

책의 그림 4-2는 **서로 다른 두 집계**를 함께 보여준다. 정산표도 둘 다 산출한다.

- **월별 확인표(monthly)**: 청산월별로 지표 계산 + 맨 아래 "평균" 행은 **달들의 평균(월 동등가중)**. 단, 총거래수 칸만 **합계**.
- **트레이딩 요약(overall)**: **전체 거래 풀링(pooled)** — 모든 왕복거래를 한 덩어리로 평균.

두 값은 서로 다르며(월평균 ≠ 전체풀링) 책도 다르게 표기한다. 검산으로 확인함(§8 참조).

### 4.2 월별 확인표 컬럼 (그림 4-2 그대로)

각 청산월 행:

| 컬럼 | 정의 |
|---|---|
| 월 | 청산 연-월 |
| 평균 수익 | 그 달 **수익거래들**의 수익률 평균(%) |
| 평균 손실 | 그 달 **손실거래들**의 손실률 평균(%, 양수로 표기) |
| 승률(%) | 그 달 수익거래 수 ÷ 그 달 총거래 수 × 100 |
| 총거래 수 | 그 달 청산된 왕복거래 수 |
| 최대 수익 | 그 달 단일 최대 수익 거래의 수익률(%) (책 스크린샷 표기 "장기 수익") |
| 최대 손실 | 그 달 단일 최대 손실 거래의 손실률(%) (책 표기 "장기 손실") |
| 수익 일수 평균 | 그 달 **수익거래들**의 평균 보유일수(정수 반올림) |
| 손실 일수 평균 | 그 달 **손실거래들**의 평균 보유일수(정수 반올림) |

**맨 아래 "평균" 행 규칙(검산 확정):**
- 총거래 수 칸 = 모든 월 값의 **합계**.
- 그 외 모든 칸 = 값이 존재하는 월들의 **산술 평균**(월 동등가중).
- **`-`(해당없음) 처리**: 그 달에 수익거래가 0이면 평균수익·최대수익·수익일수는 `-`. 손실거래가 0이면 평균손실·최대손실·손실일수는 `-`. 평균 행은 `-`인 달을 **제외하고** 평균낸다.

### 4.3 트레이딩 요약 박스 (전체 풀링, 공식 확정)

| 항목 | 공식 |
|---|---|
| 승률(성공률) | 전체 수익거래 수 ÷ 전체 왕복거래 수 × 100 |
| 평균 수익 | 전체 수익거래 수익률의 평균(%) |
| 평균 손실 | 전체 손실거래 손실률의 평균(%, 양수) |
| 성공/실패 비율 | **평균수익 ÷ 평균손실** (손익비, payoff ratio) |
| 조정 후 성공/실패 비율 | **(평균수익 × 승률) ÷ (평균손실 × 패율)** — 1보다 크면 수학적 우위 |

여기서 승률·패율은 소수(0~1). 검산: 44/95=46.32%, 13.35/5.85=2.28, (13.35×.4632)/(5.85×.5368)=1.97 — 스크린샷 값과 정확히 일치.

### 4.4 책의 추가 처방 (요약 박스 옆에 병기)

- **기대수익(expectancy)** = 승률 × 평균수익 − 패율 × 평균손실. (한 거래당 기대 수익률, %)
- **최대 수익 / 최대 손실(전체)**: 전체 기간 단일 최대 수익·손실 거래(어느 종목·언제인지 함께).
- **RBA 권장 손절폭**: `권장 최대 손절폭 = 평균수익 ÷ 손익비목표`. 손익비목표 기본 2:1 → 평균수익의 절반. 예: 평균수익 15% → 권장 7.5% 이하. 현재 `stop_loss_pct_default`(-4%)가 권장값보다 큰지/작은지 신호로 표시.
- **고집불통 트레이더 진단**(경고 플래그):
  - 최대손실 > 최대수익 → ⚠️ "손실은 붙들고 이익은 일찍 판다"
  - 손실 유지일 ≥ 수익 유지일 → ⚠️ "손실을 오래 붙들고 있다"(정상은 손실일 < 수익일)
  - `stop`이 있는 손실거래 중 `stop_violation=true` 건수 → ⚠️ "손절 규율 위반 N건"

## 6. 구성 (3개 유닛)

계산 로직을 한 곳에 두고 CLI와 페이지가 공유한다(숫자 불일치 방지). 언어는 TypeScript로 통일(A단계 Next.js 페이지가 그대로 import).

### 6.1 `src/lib/scorecard.ts` — 순수 계산 모듈 (핵심)

파일·화면을 모르는 순수 함수. 타입 정의 + 두 함수.

```ts
export type Fill = { date: string; code: string; name: string; side: "buy" | "sell";
  price: number; qty: number; fees?: number; tax?: number; stop?: number; setup?: string; note?: string };
export type Trade = { code: string; name: string; open_date: string; close_date: string;
  avg_buy: number; avg_sell: number; gross_pct: number; net_pct: number; hold_days: number;
  outcome: "win" | "loss"; month: string; buy_qty: number; sell_qty: number;
  stop?: number; stop_violation?: boolean; setup?: string };
export type OpenPosition = { code: string; name: string; qty: number; avg_buy: number; open_date: string };
export type Scorecard = { /* §5의 구조 */ };

export function matchTrades(fills: Fill[]): { trades: Trade[]; open: OpenPosition[]; errors: string[] };
export function computeScorecard(trades: Trade[], params: { rr_target: number; stop_loss_pct_default: number }): Scorecard;
```

- `matchTrades`: §3.2 규칙 구현. 오류(음수 수량 등)는 `errors`에 담아 반환(예외 던지지 않음 — CLI가 경고 출력).
- `computeScorecard`: §4 지표 전체(월별 net/gross, 요약 net/gross, RBA, 진단).

### 6.2 `scripts/build-scorecard.ts` — CLI (C단계 결과물)

- `public/data/scorecard-fills.json` 읽기 → `matchTrades` → `computeScorecard` →
  - **터미널에 정산표 출력**(월별 확인표 + 트레이딩 요약 박스 + 진단 경고). 순수익 기준 기본, 총수익 병기.
  - `errors`가 있으면 눈에 띄게 경고.
  - `public/data/scorecard.json` 저장.
- 실행: `npx tsx scripts/build-scorecard.ts` (레포에 tsx 있음, 없으면 계획 단계에서 확인).

## 7. 검증 (TDD)

계산이 전부 수학이라 단위 테스트가 필수(레포에 vitest 사용). **테스트 먼저 작성 후 구현**.

`src/lib/scorecard.test.ts` 최소 시나리오:

1. **단순 왕복**: 100주 매수 → 100주 매도, 순/총 수익률·보유일수·win 분류.
2. **분할 매수(피라미딩)**: 두 번 나눠 매수 → 한 번 매도. 가중평균 매수가 검증.
3. **분할 익절(부분 매도)**: 한 번 매수 → 두 번 나눠 매도. 가중평균 매도가·보유일수(마지막 매도일 기준).
4. **재진입 분리**: 청산 후 같은 종목 재매수 → 2개의 독립 거래.
5. **미청산 제외**: 매수만 있고 청산 안 된 종목 → `open`에만 들어가고 통계 제외.
6. **손실거래/본전거래 분류**: 순수익률 ≤ 0 → loss.
7. **월별 "-" 처리**: 어떤 달에 수익거래 0건 → 평균수익·최대수익·수익일수 null, 평균 행이 그 달 제외.
8. **요약 박스 공식**: 손으로 만든 소규모 집합에서 승률·평균수익·평균손실·성공/실패 비율·조정후 비율·기대수익이 정확히 나오는지.
9. **월별 평균 행 = 월평균, 총거래수 = 합계** 검증.
10. **수수료·세금 반영**: 순수익률이 총수익률보다 정확히 비용만큼 낮은지.
11. **엣지: 음수 수량** → `errors`에 기록, 해당 종목 제외.
12. **`stop` 규율 위반**: 손실이 계획 손절폭 초과 시 `stop_violation=true`.

가능하면 스크린샷의 12개월 집계(§8)를 하나의 통합 테스트로 재현해 공식 정합을 못박는다.
