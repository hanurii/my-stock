# KIS 자동매수 봇 v1 설계

날짜: 2026-07-07
상태: 설계 확정(사용자 승인) — 스펙 리뷰 대기
기준 브랜치: `feat/kis-autobuy` (origin/master ec262ad)

## 목적

SEPA 진입임박(actionable) 후보가 **장중 피벗을 거래량 실려 돌파**할 때 자동 매수하는 봇.
[[sepa-nextday-breakout-findings]]의 엣지(거래량·국면)를 실행에 옮긴다. "직장인이라 장을 못 본다"는
제약을 해결하되, **실자금 안전을 최우선**으로 한다.

## 범위 (v1)

- **자동매수 1주** — 조건 충족 시 종목당 **정확히 1주** 시장가 매수.
- **자동청산** — 봇이 보유 포지션을 감시하다 **−10% 손절 / +20% 목표** 도달 시 시장가 매도.
- **범위 밖(v2 이후)**: 절반익절·본전스톱·트레일링, 예산 기반 사이징(1주 초과), 클라우드 배포.

## 사용자 확정 결정

- **실제 계좌 사용**(모의투자 X). 대신 **종목당 1주만** 매수해 실거래로 테스트 → 금전 리스크 최소.
- **1주 상한은 코드에 하드코딩** — 설정으로도 못 늘림. 어떤 버그·설정 실수도 최대 1주.
- **추격 절대 금지**: 현재가가 피벗 **+3% 초과면 매수 안 함**(하드 상한 = +3%, 소프트 아님).
- 슬롯(동시 보유 상한) **10~20 설정 가능**. 종목당 항상 1주.
- 개발·검증은 **로컬**에서. 실전 배포 환경(클라우드/PC)은 검증 후 결정.
- KIS 국내주식 REST엔 서버측 자동 손절이 없어 **청산은 봇 감시 방식**(예약주문 아님).

## 구조 (부품별 — 각 독립·테스트 가능)

```
[밤] /sepa → sepa-{vcp,3c,power-play}-candidates.json (피벗 포함)
        ↓
① candidate_loader — actionable 후보(code·name·pivot·pattern) 로드 → 오늘 감시목록.
                     국면 게이트: 지수 20일선 아래면 감시목록 비움(매매 OFF).
② kis_trade (신규) — kis_api 확장: 매수/매도 주문·잔고조회·hashkey. 실전 TR.
                     핵심 안전: 주문수량 인자 없음 — 항상 1주로 하드코딩.
③ quote_poller    — 감시목록을 N초마다 현재가 조회(현재가·누적거래량 acml_vol).
④ entry_signal    — 순수 함수. 각 종목: 돌파 & 거래량pace & 추격상한 & 슬롯 & 미보유 → 매수?
⑤ exit_monitor    — 보유 포지션 감시. −10%/+20% 선착 도달 시 매도 신호(순수 판정).
⑥ safety          — 킬스위치·드라이런·일일 주문상한·슬롯상한·전량 로깅.
⑦ runner + state  — 장중 루프(09:00~15:30) 조립, 보유·주문·체결 상태 파일 유지.
```

## 판정 규칙 (④ entry_signal — 순수 함수)

입력: 종목 상태 `{price, acml_vol, pivot, avg50_vol, elapsed_frac}`, 슬롯상황, 보유셋.
```
매수 조건 (전부 충족):
  1) price >= pivot                              # 피벗 돌파
  2) price <= pivot * 1.03                        # 추격 금지(하드) — 초과면 영구 스킵
  3) vol_pace = acml_vol / (avg50_vol * elapsed_frac) >= 1.5   # 장중 거래량 pace
  4) len(open_positions) < SLOTS                 # 슬롯 여유
  5) code not in open_positions                  # 미보유(1종목 1포지션)
→ 하나라도 불충족: 스킵. (2 위반이면 그 종목은 그날 영구 스킵으로 마킹)
신호 초과 시(동시 다수): vol_pace 높은 순 우선.
```
- `elapsed_frac` = 09:00~15:30 중 경과 비율(예 10:30이면 ~0.23). 장초반 노이즈 완화 위해 09:05 이후만 판정.
- `avg50_vol`은 OHLCV 캐시(ohlcv_matrix)에서 종목별 50일 평균거래량.

## 청산 판정 (⑤ exit_monitor — 순수 함수)

입력: 보유 `{entry_price, code}`, 현재가.
```
if price <= entry_price * 0.90 → 매도(손절)
if price >= entry_price * 1.20 → 매도(목표)
```
- 손절/목표는 **진입가 기준**(1주 실체결가). 시장가 매도.
- (v1은 30일 타임스톱·트레일링 없음 — v2.)

## KIS 주문 모듈 (② — 신규, 안전 최우선)

- kis_api에 추가: `place_buy_1share(code)`, `place_sell_1share(code)`, `inquire_balance()`, `_hashkey()`.
- **주문 수량 매개변수 없음** — 함수 내부에서 `ORD_QTY="1"` 하드코딩. 1주 초과 주문은 코드상 불가능.
- 실전 TR(구현 시 KIS Developers 문서·koreainvestment/open-trading-api 로 정확 확인):
  매수 `TTTC0802U`, 매도 `TTTC0801U`, 잔고 `TTTC8434R`, 해시키 `/uapi/hashkey`. `KIS_ENV=real`.
- 초당 호출 20건 제한 준수(기존 `_throttle` 활용/확장).

## 안전장치 (⑥ — 실자금 필수)

- **모드 3단**: `dryrun`(로그만, 주문 X) → `live`(실주문). 기본 dryrun. live 전환은 설정+실행인자 둘 다 필요.
- **1주 하드코딩**(②)로 최대 손실 = 종목당 1주.
- **킬스위치**: `KILL` 파일 존재 시 즉시 모든 신규 주문 중단(보유 청산 감시는 유지 옵션).
- **일일 상한**: 최대 신규주문 수(=슬롯), 도달 시 정지. 중복(같은 code) 재매수 금지.
- **전량 로깅**: 모든 판정·주문요청·응답·거부·에러 → 파일 로그 + 일일 요약.
- **장 시간 가드**: 09:05~15:20만 신규매수, 15:20 이후는 청산만.

## 설정 (config, 조정 가능)

- `SLOTS`(기본 10, 10~20), `VOL_PACE_MIN`=1.5, `CHASE_MAX_PCT`=3.0(하드), `TARGET_PCT`=20, `STOP_PCT`=10,
  `POLL_SEC`=3~5, `REGIME_FILTER`=on(지수>20MA), `MODE`=dryrun/live.
- **수량은 config에 없음**(항상 1주).

## 데이터 흐름

밤: /sepa → 후보 JSON. 아침(개장 전): 국면 체크 → 감시목록 확정. 장중: 폴링→④진입·⑤청산→②주문.
장마감: 일일 요약 로그.

## 테스트

- **④ entry_signal / ⑤ exit_monitor: 순수 함수 → pytest**(합성 상태로 돌파·추격상한·pace·슬롯·손절·목표 전 케이스).
- **② kis_trade: 실계좌 1주 스모크**(가장 유동적 소액주 1주 매수→즉시 매도로 주문경로 검증). dryrun 먼저.
- **⑦ runner: dryrun 통합**(하루 로그만 뽑아 신호·판정 육안 검증) → 이후 live 1주.

## 개발 순서 (구현 태스크)

1. entry_signal 순수함수(TDD) 2. exit_monitor 순수함수(TDD) 3. candidate_loader + 국면 게이트
4. kis_trade 주문/잔고(1주 하드코딩, dryrun+실계좌 스모크) 5. quote_poller(현재가·누적거래량)
6. runner + state + safety(킬스위치·상한·로깅) 7. dryrun 통합 실행

## 한계·주의

- 거래량 pace는 "당일 마감 거래량"의 장중 근사(오전 폭증형은 잡고 오후 늦게 몰리는 건 놓칠 수 있음).
- 봇 감시 청산은 폴링 간격·네트워크에 의존(장중 급락 시 −10%보다 더 빠져 체결될 수 있음 = 슬리피지).
- 실계좌 1주라도 **실주문**이다 — dryrun 충분히 검증 후 live. API 다운·부분체결·중복주문 방어 필수.
- 미래에셋 서버측 자동주문 등 대안은 v1 범위 밖(정밀 거래량 게이트 때문에 KIS API 채택).
